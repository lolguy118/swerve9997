# Author the design-review loop for this repo

You are a principal product engineer and architect. Running this command **once** authors a turnkey
definition-and-design loop for THIS repo, then stops. You CREATE the loop's files; you DO NOT launch it —
launching is a separate, explicit `/design-review-loop-run` (or `ralph-loop`) invocation.

This is a **doc sub-family** loop. **Follow the shared authoring protocol in
[`../loop-authoring-protocol.md`](../loop-authoring-protocol.md)** (3-tier input precedence, repo audit,
frozen taxonomy, the four-file write, phases A/C/E/F, and the universal HARD RULES). This file supplies only
the **deltas** below. It is the **upstream feeder** for the doc- and impl-review loops: the corpus it
produces (a product brief, ADRs, design docs) is the spec-of-record those loops consume.

## Delta 1 — the lens is captured by a FRONT-LOADED INTERVIEW

The protocol's Step 1 resolves the lens from `$ARGUMENTS`, else the pre-filled `<PRODUCT_VISION>` block,
else the owner. For this loop, "ask the owner" means **conduct a short front-loaded interview now** — do not
settle for a single menu. "What should this product be?" needs human judgment up front; the loop must not
invent a product vision.

<PRODUCT_VISION>

**The interview (menu path).** First ingest whatever inputs exist: skim any existing repo, notes, brief, or
files the owner points at. Then interview the owner **one focused question at a time** (use `AskUserQuestion`
for the structured choices; plain prose for open ones), covering, in roughly this order:

1. **Purpose / problem** — what does this product do, and what problem does it solve?
2. **Audience** — who is it for; who are the primary users / buyers?
3. **Scope & non-goals** — what is in v1, and what is explicitly out?
4. **Success** — how will success be measured (the few metrics that matter)?
5. **Constraints** — budget, timeline, platform, regulatory, team, or technical constraints.
6. **Major decisions** — the handful of consequential choices already made or that must be made (stack,
   architecture shape, build-vs-buy, data model, key integrations).

Synthesize the answers into a **seed product brief + a decision list**, echo both back, and have the owner
confirm or correct them before you derive the taxonomy. Treat the confirmed brief as `<PRODUCT_VISION>` for
the rest of this command. If the owner cannot answer a foundational question yet, record it as an
`OPEN QUESTION (needs owner)` rather than guessing.

## Delta 2 — taxonomy: the design corpus, in dependency order

From the brief + decision list + audit, derive the priority-ordered, dependency-ordered set of **design
artifacts** the product needs:

- **product brief** sections (purpose, audience, scope, success, constraints) — the foundation;
- **one ADR per major decision**, dependency-ordered (a decision that rests on an earlier one comes after
  it); scaffold each with this repo's `new-adr` template (`docs/ADRs/_template.md`) when present;
- **design docs** for the subsystems the decisions imply.

Give each artifact a one-line **done-bar** (what it must answer to count as done). Output locations: the
brief under `docs/product/` (create it if absent), ADRs under `docs/ADRs/`, design docs under `docs/design/`.

## Delta 3 — phase B (Sweep) and phase D (Advance)

- **B. Sweep** — re-scan the inputs + the emerging corpus for **newly-implied decisions** (a written ADR
  often surfaces the next one). Append each as a new ledger item.
- **D. Advance** — create or advance **exactly one artifact per pass** to its done-bar. When a pass
  hits a genuine product judgment call you cannot derive from the brief, **do not invent it** — record it as
  `OPEN QUESTION (needs owner)` and either advance a different non-blocked artifact or end the pass. Never
  rewrite an artifact already at its done-bar.

## Delta 4 — gate (doc sub-family): the liability self-guard

Embed this rule in the driver verbatim, as a HARD RULE after the universal four:

> Every legal / financial / compliance / tax / business-model artifact the loop emits MUST begin, right
> after its H1, with this blockquote, verbatim:
>
> > **DRAFT — NOT legal or financial advice. Generated by an automated loop; requires professional and
> > human review before any reliance.**
>
> Such artifacts must also state their key assumptions explicitly so a human reviewer can check them.

## Delta 5 — convergence and the completion phrase

Convergence (per the protocol's universal rule 3) additionally requires that **no `OPEN QUESTION (needs
owner)` blocks a still-unbuilt foundational artifact** — parked questions on non-blocking artifacts may
remain. The driver's Phase-F emission uses the completion phrase `DESIGN REVIEW LOOP COMPLETE`.

## Then print the launch line and stop

Print this for the owner (one line), then tell them the loop is written but NOT running:

```text
/ralph-loop:ralph-loop "Define and design the product per docs/planning/ralph/design-review-loop.prompt.md" --max-iterations 40 --completion-promise 'DESIGN REVIEW LOOP COMPLETE'
```

End with: "The loop is written and committed-ready, but it is NOT running. Run `/design-review-loop-run`
(or the command above) to start it; stop it early with `/ralph-loop:cancel-ralph`."
