# ADR-009: JUnit 5 + JaCoCo + HAL Simulation for Library Tests

## Status

Accepted

## Date

2026-04-20

## Context

The library wraps hardware devices (TalonFX, CANcoder, Pigeon 2, etc.)
that cannot exist on a CI runner. WPILib provides a Hardware
Abstraction Layer (HAL) with a simulation mode — when initialized,
device objects become functional stand-ins without real hardware.

Robot code also depends on `CTREManager` static state that persists
across JVM lifetime. Ordinary JUnit tests would pollute this state
between tests within the same class run.

## Decision

The library uses **JUnit 5 Jupiter** for test authoring, **JaCoCo** for
coverage, and **WPILib HAL simulation** for hardware-touching tests.
Every test class that touches hardware initializes HAL once with
`@BeforeAll` and resets `CTREManager` state in `@BeforeEach`:

```java
@BeforeAll
static void initHAL() {
    HAL.initialize(500, 0);
}

@BeforeEach
void setup() {
    CTREManager.resetForTesting();
}
```

`libtest/` is a separate source set containing integration-style
harnesses (`Infrastructure`, `Superstructure`) and is excluded from
coverage metrics.

## Rationale

1. **JUnit 5 language features.** Parameterized tests, nested classes,
   `@BeforeEach`/`@BeforeAll` are natural fits for testing a lifecycle
   library.
2. **HAL simulation is free.** WPILib ships HAL sim as part of
   standard GradleRIO; no additional test framework is needed.
3. **CTRE SimState integrates cleanly.** With HAL initialized, CTRE
   devices expose functional `SimState` objects — `simulationInit()`
   and `setSimPosRotations()` can be verified without real hardware.
4. **`resetForTesting()` over reflection.** `CTREManager` exposes an
   explicit reset method; reflection-based teardown (older pattern)
   still works but is no longer necessary.
5. **JaCoCo is the FRC standard.** Coverage reports are generated by
   GradleRIO out of the box; adoption aligns with what students will
   encounter in other FRC codebases.

## Consequences

**Easier:**

- Tests run on CI without hardware.
- Students can run tests locally without a robot.
- Coverage reports highlight untested paths in library code.

**Harder:**

- Tests that create CTRE devices must remember HAL init + CTREManager
  reset. Forgetting leads to cryptic native-library errors or cross-
  test pollution.
- Unique CAN IDs are required across tests in the same class to avoid
  device conflicts.
- Some behaviors (real CAN timing, actual motor output, firmware
  closed-loop response) cannot be verified in unit tests and must be
  validated on hardware.

## Alternatives Considered

- **JUnit 4.** Rejected — no autodetection for extensions, weaker
  parameterization, dated.
- **TestNG.** Rejected — not used elsewhere in FRC; no advantage for
  our patterns.
- **No HAL sim — mock every hardware call.** Rejected — mocks would
  diverge from real behavior over time; HAL sim is authoritative.
